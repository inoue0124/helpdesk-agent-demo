# ヘルプデスクエージェント デモシステム — 要件定義書

## 1. プロジェクト概要

### 目的

問い合わせ対応エージェントを組み込んだデモシステムを構築する。

### デモで見せるもの

- ユーザーがお問い合わせフォームから質問を送信する画面
- エージェントが回答ドラフトをリアルタイムに生成する過程（SSE）
- オペレーターが回答を確認・編集・承認して送信する画面

### 技術的なアピールポイント

| 要素 | 実装内容 |
|---|---|
| AI/LLM 活用 | LangGraph エージェント、リフレクション付き自動回答生成 |
| バックエンド設計 | FastAPI, SSE による非同期リアルタイム通信 |
| フロントエンド | Next.js (App Router), エージェント処理過程のリアルタイム可視化 |
| 検索技術 | Elasticsearch (全文検索) + Qdrant (ベクトル検索) のハイブリッド |
| インフラ | Docker Compose でワンコマンド起動、Caddy で自動HTTPS |
| 実務想定 | 信頼度ベースのルーティング、Human-in-the-loop、フィードバックループ |

### Dify との差別化 — なぜ自作するのか

Dify はナレッジ管理・RAG パイプラインが優秀だが、以下は Dify では実現できない。
本プロジェクトはこれらに注力する。

| Dify にないもの | 本プロジェクトでの実装 |
|---|---|
| エージェント処理過程のリアルタイム可視化 | SSE で各ノード（計画→検索→回答→リフレクション）をライブ表示 |
| リフレクション付きリトライの可視化 | LangGraph のサイクルグラフで自己評価→再検索→再回答の過程を表示 |
| 信頼度ベースの自動ルーティング | 高信頼→自動承認、中信頼→オペレーター確認、低信頼→エスカレーション |
| 適応的検索戦略 | エージェントが質問内容に応じて ES / Qdrant を動的に選択 |
| オペレーター向け承認ダッシュボード | 問い合わせ一覧、進捗可視化、回答編集、承認フローを統合した UI |

## 2. 機能要件

### 2.1 ユーザー問い合わせフォーム (`/user`)

- お問い合わせフォーム形式で質問を送信できる
- フォーム項目: ユーザー名（任意）、問い合わせ内容（必須）
- 送信後に受付完了メッセージを表示する
- 回答が届いたら画面上に表示する（ポーリングまたは SSE）

### 2.2 オペレーターダッシュボード (`/operator`)

- 問い合わせ一覧をステータス別にフィルタして表示できる
- 問い合わせ詳細画面でエージェント処理の進捗をリアルタイムに確認できる（SSE）
- エージェント処理の各ステップ（計画→ツール選択→検索→回答→リフレクション→リトライ→統合）を可視化する
- 回答ドラフトを編集できる
- 承認/却下ボタンで回答を送信できる
- 信頼度バッジを表示する

### 2.3 LangGraph エージェント

- 質問をサブタスクに分解する（planner）
- サブタスクごとに最適な検索ツールを選択する（tool_selector）
- Elasticsearch / Qdrant で検索を実行する（searcher）
- 検索結果をもとに回答を生成する（answerer）
- 回答品質を自己評価し、不十分なら別クエリでリトライする（reflector、最大3回）
- サブタスク結果を統合して最終回答を生成する（integrator）

### 2.4 信頼度ベースのルーティング

| 信頼度 | アクション | ステータス |
|---|---|---|
| >= 0.9 | 自動承認。回答をユーザーに即送信 + QA蓄積 | `auto_approved` |
| 0.5 〜 0.89 | オペレーターに確認を依頼 | `draft` |
| < 0.5 | エスカレーション（上位担当者に通知） | `escalated` |

### 2.5 フィードバックループ

- 承認された回答（`approved` / `auto_approved`）は QA ペアとして自動蓄積する
- SQLite の `qa_pairs` テーブルに保存する
- Qdrant にベクトル登録し、次回以降の類似質問検索に活用する

### 2.6 レート制限

- 1日あたりの問い合わせ上限を設ける（デフォルト: 50件/日）
- 上限到達時は 429 エラーを返す

## 3. 非機能要件

| 項目 | 要件 |
|---|---|
| レスポンス | 問い合わせ受付は 1 秒以内。エージェント処理は SSE で進捗を表示 |
| 可用性 | デモ用途のため SLA なし。Docker Compose でワンコマンド復旧 |
| セキュリティ | 認証なし（デモ用）。CORS 許可。OpenAI API キーはサーバーサイドのみ |
| コスト | OpenAI API の日次上限で制御 |
| デプロイ | Hetzner Cloud CAX21 (ARM / 4vCPU / 8GB / 80GB) 月額約 7 EUR |
| 日本語対応 | Elasticsearch は kuromoji プラグインで日本語形態素解析 |

## 4. スコープ外

以下は本プロジェクトでは実装しない:

- ナレッジ管理画面（Dify の得意領域）
- ドキュメントアップロード UI
- チャンク分割の設定 UI
- セグメント単位の編集 UI
- ユーザー認証・認可
- マルチテナント対応

## 6. ナレッジ管理方針

### 実装するもの

| 機能 | 方法 | 理由 |
|---|---|---|
| 初期データ投入 | CLI スクリプト `python -m scripts.create_index` | 初回セットアップで十分 |
| マニュアル更新 | ファイル差し替え → スクリプト再実行 | 頻度が低い。UI は不要 |
| QA 自動蓄積 | 承認時に SQLite + Qdrant へ自動登録 | フィードバックループとして実装 |

### 実装しないもの

- ナレッジ管理画面
- ドキュメントアップロード UI
- チャンク分割の設定 UI
- セグメント単位の編集 UI
