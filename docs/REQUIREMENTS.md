# AI Agent Demo Portfolio — 要件定義書

## 1. プロジェクト概要

### 目的

4 種類の AI エージェントを実際に試せるデモサイトを構築する。
各エージェントは異なるユースケース・アーキテクチャパターンを実装し、LangGraph ベースのエージェント技術を体系的に示す。

### デモで見せるもの

- ポータル画面からエージェントを選択して試せる統合的な体験
- 各エージェントの処理過程をリアルタイムに可視化（SSE）
- エージェントごとに異なる LangGraph パターンの実装

### 技術的なアピールポイント

| 要素 | 実装内容 |
|---|---|
| AI/LLM 活用 | 4 種類の LangGraph エージェント（Reflection, Code Gen, Multi-Agent, Multi-Chain） |
| バックエンド設計 | FastAPI, SSE による非同期リアルタイム通信 |
| フロントエンド | Next.js (App Router), エージェント処理過程のリアルタイム可視化 |
| 検索技術 | Elasticsearch + Qdrant のハイブリッド（問い合わせエージェント） |
| コード実行 | E2B サンドボックスによる安全なコード実行（データ分析エージェント） |
| インフラ | Docker Compose でワンコマンド起動 |

## 2. エージェント別要件

### 2.1 問い合わせ対応エージェント

- ユーザーがフォームから問い合わせを送信する
- エージェントが質問をサブタスクに分解し、検索→回答→リフレクションの流れで処理する
- リフレクションで不十分な場合はリトライする（最大 3 回）
- 信頼度に基づいて自動承認 / オペレーター確認 / エスカレーションにルーティングする
- オペレーターが回答を確認・編集・承認する
- 承認された回答は QA ペアとして自動蓄積する（フィードバックループ）

### 2.2 データ分析エージェント

- ユーザーが CSV ファイルをアップロードし、分析リクエストを入力する
- エージェントが分析計画を立て、Python コードを生成・実行する
- コード実行は E2B サンドボックスで安全に行う
- 実行結果をレビューし、不十分なら再生成する
- 分析結果（グラフ・テーブル・インサイト）をレポートとして出力する

### 2.3 マーケティング支援エージェント

- ユーザーがマーケティング課題を入力する
- 複数の専門エージェントが協調して対応する
  - Router: 質問内容を分類し適切なエージェントに振り分け
  - Question: 状況を深掘りする質問を生成
  - Recommendation: 具体的な施策を提案
- 会話形式でインタラクティブにやり取りする

### 2.4 提案資料作成エージェント

- ユーザーがテーマ・目的を入力する
- エージェントがヒアリングで要件を明確化する
- Web 検索で情報を収集する
- 収集情報を評価し、不足していればリトライする
- 最終的に構造化されたレポート（Markdown）を生成する

## 3. 共通要件

### ポータル画面

- 4 つのエージェントを選択できるランディングページ
- 各エージェントの概要・ユースケース・技術的特徴を説明する
- エージェントごとに専用の画面に遷移する

### リアルタイム可視化

- 全エージェントで処理過程を SSE でリアルタイム表示する
- 各ノードの実行状況（計画→実行→評価）を可視化する

### レート制限

- 全エージェント共通で 1 日あたりの API 呼び出し上限を設ける

---

## 4. 非機能要件

| 項目 | 要件 |
|---|---|
| レスポンス | リクエスト受付は 1 秒以内。処理は SSE で進捗表示 |
| セキュリティ | 認証なし（デモ用）。API キーはサーバーサイドのみ |
| コスト | OpenAI API の日次上限で制御 |
| デプロイ | Hetzner Cloud で Docker Compose 構成 |
| コード実行 | E2B サンドボックスで隔離（データ分析エージェント） |

---

## 5. スコープ外
