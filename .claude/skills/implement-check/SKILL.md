# 実装 vs 設計の準拠チェック

フィーチャー: $ARGUMENTS

## 手順

1. `docs/features/<feature>/DESIGN.md` を読む
2. `docs/features/<feature>/TASKS.md` を読み、`- [x]` の完了タスクを確認する
3. 完了タスクに対応する実装ファイルを全て読む
4. 以下の観点で設計との差異をレポートする:

### チェック観点

| # | 観点 | 確認内容 |
|---|------|----------|
| 1 | ファイル配置 | DESIGN.md のファイル一覧と実ファイルが一致しているか |
| 2 | API シグネチャ | エンドポイント、リクエスト/レスポンス型が設計通りか |
| 3 | データモデル | テーブル定義、Pydantic モデルが設計通りか |
| 4 | SSE イベント | `docs/DESIGN.md` セクション 5.1 のイベントコントラクトに準拠しているか |
| 5 | エラーハンドリング | `AppError` を使っているか、エラーコードが一覧に含まれているか |
| 6 | 外部サービス呼び出し | `with_retry` を通しているか、タイムアウト値が設計通りか |
| 7 | DI | `Depends()` で注入しているか、グローバル変数を使っていないか |

### 出力フォーマット

```
## 準拠チェック結果: <フィーチャー名>

### OK
- <準拠している項目>

### 差異あり
- <ファイル:行番号> — <設計との差異の説明>

### 未実装
- <DESIGN.md に記載があるが未実装の項目>
```
